# Story: Watch config files and hot-reload on change

## User Story

**As an** operator,
**I want** the Clincus web server to detect changes to system and user config files and reload automatically,
**so that** I don't have to manually restart the server after editing configuration.

---

## Context

Config is loaded once at startup via `config.Load()` in `root.go`. There is no reload mechanism. The server runs `http.ListenAndServe()` with no graceful shutdown support. This story adds file watching on `/etc/clincus/config.toml` and `~/.config/clincus/config.toml`, and implements in-process hot-reload: re-read config, swap the server's config reference, and briefly cycle the HTTP listener if the port changed.

Active sessions are Incus containers running tmux — they are fully independent of the web server process. A reload may briefly disconnect WebSocket terminal connections, but tmux sessions continue and clients can reconnect.

---

## Acceptance Criteria

- [ ] AC1: The server watches `/etc/clincus/config.toml` and `~/.config/clincus/config.toml` for modifications using filesystem notifications (e.g., fsnotify)
- [ ] AC2: When a watched file changes, the server re-runs `config.Load()` and updates its internal config reference
- [ ] AC3: If the dashboard port has not changed, active HTTP connections and WebSocket sessions are preserved (no listener restart needed)
- [ ] AC4: If the dashboard port changes, the HTTP listener is cycled: graceful shutdown of the old listener, start on the new port
- [ ] AC5: Active Incus container sessions are unaffected by any reload — verified by test or documented validation
- [ ] AC6: Config reload is debounced (e.g., 1-second window) to handle editors that write files in multiple steps
- [ ] AC7: Reload errors (invalid TOML, missing file) are logged but do not crash the server — the previous valid config is retained
- [ ] AC8: Non-existent watched paths are handled gracefully (file may not exist at startup but should be picked up if created later)
- [ ] AC9: Existing Go tests pass; new tests cover the file watching and reload logic

---

## Out of Scope

- Watching project-level config (`.clincus.toml`) or `CLINCUS_CONFIG` env var path
- Full process restart (systemd, supervisor)
- Config validation beyond TOML parsing

---

## Dependencies

- None (can be implemented independently of E01S01/E01S02, but E01S04 consumes its reload signal)

---

## Notes

The `fsnotify` package is the standard Go library for cross-platform file watching. Debouncing is important because editors like vim write to a temp file then rename, producing multiple events. The server needs a `sync.RWMutex` or similar to protect concurrent config reads during reload. Consider adding a `Reload() error` method to the `Server` struct.
