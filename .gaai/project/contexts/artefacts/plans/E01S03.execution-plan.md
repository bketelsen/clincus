# Execution Plan: E01S03 — Watch config files and hot-reload on change

## Tier: 2 (Core Team)

## Approach

Use `github.com/fsnotify/fsnotify` — the standard Go file-watching library. Debounce via `time.AfterFunc` with 1-second window. Thread-safe config swap via `sync.RWMutex`. Graceful HTTP listener restart via `http.Server.Shutdown()` + re-listen when port changes.

No approach evaluation needed — fsnotify is the only production-viable cross-platform option for Go filesystem notifications.

## Implementation Sequence

### Step 1: Add fsnotify dependency

- `go get github.com/fsnotify/fsnotify`

### Step 2: Create `internal/config/watcher.go`

New file. Contains:
- `Watcher` struct: holds `fsnotify.Watcher`, debounce timer, reload callback, logger
- `WatchPaths()` — returns system + user config paths (not project config, per out-of-scope)
- `NewWatcher(paths []string, reloadFn func() error) (*Watcher, error)` — creates watcher
  - For each path: if file exists, watch it; if not, watch parent directory (AC8)
  - Handle fsnotify events: on Write/Create/Rename, debounce and call reloadFn
- `(*Watcher).Start()` — starts event loop goroutine
- `(*Watcher).Close()` — stops watcher
- Debouncing: 1-second `time.AfterFunc` reset on each event (AC6)
- Non-existent paths: watch parent directory, detect file creation (AC8)

### Step 3: Create `internal/config/reload.go`

New file. Contains:
- `ConfigManager` struct: holds `*Config`, `sync.RWMutex`, watcher reference
- `NewConfigManager() (*ConfigManager, error)` — loads initial config, starts watcher
- `(*ConfigManager).Get() *Config` — read-locked config access
- `(*ConfigManager).Reload() error` — re-runs `Load()`, swaps config under write lock
  - On error: log, retain previous config (AC7)
  - Returns old config for comparison (port change detection)
- `(*ConfigManager).Close()` — stops watcher

### Step 4: Modify `internal/server/server.go`

- Add `configMgr *config.ConfigManager` field to `Server` struct (or accept callback)
- Add `OnConfigReload(oldCfg, newCfg *config.Config)` method
  - If port changed: signal listener restart
  - If port unchanged: update in-place (AC3)

### Step 5: Modify `internal/cli/serve.go`

- Replace direct `config.Load()` usage with `config.NewConfigManager()`
- Wire reload callback to server
- Replace `http.ListenAndServe` with `http.Server` for graceful shutdown support (AC4)
- Implement listener cycling loop: start server, on port change signal, shutdown + restart

### Step 6: Add tests

- `internal/config/watcher_test.go`:
  - Test: file modification triggers reload callback
  - Test: debouncing (rapid writes produce single reload)
  - Test: non-existent file path handled gracefully
  - Test: invalid TOML does not crash (AC7)
- `internal/config/reload_test.go`:
  - Test: ConfigManager.Get() returns current config
  - Test: ConfigManager.Reload() swaps config
  - Test: Reload with invalid TOML retains old config

### Step 7: Update documentation

- `docs/reference/config.md` — add section on hot-reload behavior
- `docs/guides/web-dashboard.md` — mention auto-reload capability

## Edge Cases

- AC6: Editors like vim write temp file then rename — produces multiple fsnotify events. Debounce window absorbs these.
- AC7: Invalid TOML after edit — logged as warning, old config retained.
- AC8: Config file does not exist at startup but created later — watch parent dir, pick up Create event.
- AC4: Port change during active WebSocket sessions — old connections drop on shutdown, clients reconnect to new port.
- AC5: Incus containers run independently — reload only affects HTTP server process, no container impact.

## Risk Register

- **Low:** fsnotify may not detect changes on network filesystems — acceptable, config files are local
- **Low:** Race between concurrent reloads — mitigated by debounce + mutex
